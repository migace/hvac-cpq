Detect model changes and generate an Alembic migration for the backend database.

## Context

- Backend: `apps/backend/`
- Models: `apps/backend/app/db/models.py`
- Alembic config: `apps/backend/alembic.ini`
- Alembic env: `apps/backend/alembic/env.py`
- Migrations: `apps/backend/alembic/versions/`
- Database URL comes from `apps/backend/app/core/config.py` (Settings, loaded from `.env`)

## Steps

### 1. Analyze changes

- Read `apps/backend/app/db/models.py` to understand current model state.
- Run `git diff apps/backend/app/db/models.py` to see what changed.
- If no model changes exist, check `git diff main...HEAD -- apps/backend/app/db/models.py` for branch changes.
- List existing migrations: `ls apps/backend/alembic/versions/` and read the latest one to understand current schema state.

### 2. Describe the migration

Before generating, explain to the user:
- What model changes were detected
- What SQL operations the migration will perform (CREATE TABLE, ADD COLUMN, ALTER COLUMN, CREATE INDEX, DROP, etc.)
- Whether the migration is reversible (downgrade path)
- Any potential data loss risks (dropping columns, changing types)

### 3. Generate the migration

Run from `apps/backend/`:

```bash
cd apps/backend && python -m alembic revision --autogenerate -m "description_of_changes"
```

The message should be snake_case, descriptive, under 60 characters. Examples:
- `add_product_quotes`
- `add_blade_type_to_attributes`
- `create_audit_log_table`

### 4. Review generated migration

- Read the generated migration file in `apps/backend/alembic/versions/`.
- Verify:
  - `upgrade()` contains the expected operations
  - `downgrade()` properly reverses all changes
  - No unintended changes were picked up (Alembic autogenerate can be noisy)
  - Indexes are created for foreign keys and frequently queried columns
  - CASCADE delete is set where appropriate (follow existing patterns)
  - Enum types are handled correctly if applicable

### 5. Clean up if needed

- Remove any autogenerated operations that don't belong (sometimes Alembic detects false diffs).
- Ensure the migration imports match what's needed (`op`, `sa`, etc.).

### 6. Test the migration

If a database is available, run:

```bash
cd apps/backend && python -m alembic upgrade head
cd apps/backend && python -m alembic downgrade -1
cd apps/backend && python -m alembic upgrade head
```

This verifies both upgrade and downgrade work.

### 7. Show SQL preview

Run to show raw SQL:

```bash
cd apps/backend && python -m alembic upgrade head --sql
```

## Important

- NEVER modify existing migration files — only create new ones
- NEVER delete migration files
- If autogenerate produces an empty migration, inform the user that models are in sync with migrations
- Warn about destructive operations (DROP COLUMN, DROP TABLE, type changes) — these may cause data loss
- If the migration involves dropping or renaming, suggest a two-step approach (add new → migrate data → drop old)
- Always ensure downgrade() is functional — no-op downgrades are acceptable only for truly irreversible operations (with a comment explaining why)
